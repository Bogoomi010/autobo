# 자동화 프롬프트: 틱 단위 매수 매도 전략

## 제목

틱 단위 매수 매도 전략

## 내용

해당 프로젝트에서 24시간 매수 매도를 진행하면서 어떻게 해야 더 많은 수익을 낼 수 있을지 알고리즘 또는 투자 전략을 연구하고 기획합니다.

이전 자동화 메모리와 `docs` 문서를 먼저 읽고, 이미 진행한 기능 구현 작업과 겹치지 않는 연구 범위를 선정합니다. 특히 기존의 단순 기준가 자동 전략, 마켓 즐겨찾기, 수동 주문 프리셋, 화면 설정 저장 기능과 중복되지 않도록 합니다.

연구 및 기획은 텍스트 파일로 기록합니다. 코드 구현은 진행하지 않고, 전략 설계와 검증 계획을 문서로만 남깁니다.

## 연구 우선순위

1. 틱 단위 가격 변동에서 반복적으로 관찰 가능한 매수/매도 신호
2. 수수료, 슬리피지, 호가 스프레드, 미체결 리스크를 반영한 기대수익 구조
3. 24시간 운용 시 손실 제한, 거래 빈도 제한, 쿨다운, 일일 중단 조건
4. 실거래 전 백테스트와 모의매매로 검증 가능한 알고리즘 형태
5. 프로젝트의 기존 Upbit API 경계와 안전 원칙을 지키는 실행 구조

## 필수 검토 항목

- Upbit 틱, 체결, 호가, 캔들 데이터 중 어떤 데이터를 전략 입력으로 사용할지 비교합니다.
- 시장가, 지정가, 최유리 지정가 주문의 장단점을 틱 단위 전략 관점에서 정리합니다.
- 수수료와 최소 주문 금액을 고려했을 때 유의미한 가격 변동 폭을 계산하는 방법을 제안합니다.
- 과도한 매매로 수수료가 누적되는 상황을 방지하는 조건을 설계합니다.
- 횡보장, 급등장, 급락장, 저유동성 장에서 전략이 어떻게 달라져야 하는지 정리합니다.
- 실거래 주문을 바로 실행하지 않고 dry-run, 백테스트, 모의매매 단계로 검증하는 계획을 포함합니다.

## 산출물

`docs` 경로에 전략 연구 및 기획 결과 텍스트 파일을 생성합니다.

문서에는 다음 내용을 포함합니다.

- 전략 목표
- 사용 데이터
- 매수 신호
- 매도 신호
- 포지션 크기와 리스크 제한
- 수수료와 슬리피지 반영 방식
- 백테스트 또는 모의매매 검증 계획
- 실거래 연결 전 필요한 안전장치
- 구현 시 필요한 코드 변경 범위

## 제한 사항

- 실제 매수/매도 주문을 실행하지 않습니다.
- API Key, 계정 정보, 주문 응답 등 민감 정보를 문서에 기록하지 않습니다.
- 수익을 보장하는 표현을 사용하지 않고, 전략의 실패 조건과 손실 가능성을 함께 기록합니다.
- 입금과 출금 API는 전략 연구 범위에서 제외합니다.

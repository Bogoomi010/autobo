# 틱 단위 매수 매도 전략 기획

작성일: 2026-05-18

## 전략 목표

Autobo의 기존 자동 전략은 현재가가 기준가 이하 또는 이상에 도달하면 시장가 주문을 만드는 단순 조건식이다. 이 문서는 그 기능과 겹치지 않게, 틱 단위 체결 흐름과 호가 상태를 입력으로 사용하는 단기 매매 전략을 연구하고 검증하기 위한 설계 초안을 정리한다.

목표는 24시간 자동 운용에서 반복 가능한 짧은 구간의 우위가 있는지 검증하는 것이다. 수익을 보장하지 않으며, 수수료, 슬리피지, 스프레드, 미체결, 급변장 손실 때문에 기대값이 쉽게 음수가 될 수 있다는 전제를 둔다.

## 사용 데이터

### 1. WebSocket trade

실시간 전략 입력의 기본 데이터다. REST `GET /v1/trades/ticks`는 최근 체결 조회와 백테스트 샘플 수집에 사용하고, 실시간 판단에는 Upbit WebSocket `trade` 구독을 우선한다.

사용 필드:

- 마켓 코드
- 체결 가격
- 체결 수량
- 체결 시각
- 매수/매도 체결 방향을 식별할 수 있는 필드

전략 입력:

- 최근 N초 체결 수
- 최근 N초 체결대금
- 매수 주도 체결대금과 매도 주도 체결대금의 불균형
- 직전 틱 대비 가격 변화 방향
- 같은 방향 틱이 연속된 횟수

### 2. WebSocket orderbook

틱 전략은 호가 스프레드와 잔량 변화에 민감하다. 실시간 실행 단계에서는 `orderbook` 구독을 추가해서 매수/매도 호가 최상단의 가격과 잔량을 함께 본다.

사용 필드:

- 최우선 매수호가와 매도호가
- 최우선 호가 잔량
- 상위 K단계 호가 잔량 합
- 스프레드

전략 입력:

- 매수호가 잔량 대비 매도호가 잔량 비율
- 스프레드가 목표 기대수익보다 큰지 여부
- 주문 수량이 상위 호가 잔량에 비해 과도한지 여부
- 호가가 빠르게 비는 상황의 추적

### 3. ticker와 candle

`ticker`는 전일 대비 등락률, 24시간 거래대금, 현재가 화면 표시와 마켓 필터링에 사용한다. `candle`은 단기 추세 필터와 백테스트 보조 지표에 사용한다. 다만 실시간 진입/청산 판단의 1차 입력은 체결과 호가로 둔다.

## 핵심 신호

### 매수 후보 신호

다음 조건이 동시에 가까워질 때만 매수 후보로 분류한다.

- 최근 `windowSec` 동안 매수 주도 체결대금 비율이 `buyImbalanceThreshold` 이상이다.
- 직전 `momentumTicks`개 틱 중 상승 틱 비율이 `upTickRatioThreshold` 이상이다.
- 최우선 매도호가 잔량이 주문 예정 수량보다 충분히 크고, 스프레드가 허용 범위 이하다.
- 최근 `cooldownSec` 안에 같은 마켓에서 매수 또는 손절 주문을 내지 않았다.
- 1분 또는 3분 캔들 기준 단기 변동성이 `minVolatilityRate` 이상이며, 너무 낮은 횡보장 필터에 걸리지 않는다.

매수 후보 점수 예시:

```text
buy_score =
  0.35 * trade_imbalance_score
+ 0.25 * uptick_momentum_score
+ 0.20 * orderbook_support_score
+ 0.10 * liquidity_score
+ 0.10 * volatility_score
```

`buy_score`가 기준값 이상이어도 즉시 실주문하지 않는다. 최초 구현은 dry-run 이벤트로 기록하고, 검증 단계에서 기대값이 양수인 조건만 주문 후보로 승격한다.

### 매도 후보 신호

매도는 익절, 손절, 시간 만료, 신호 반전으로 분리한다.

- 익절: 진입 평균가 대비 예상 순수익률이 목표값 이상이고, 매도 시점의 스프레드와 예상 슬리피지를 반영해도 목표가 유지된다.
- 손절: 진입 후 가격이 `stopLossRate` 이상 역방향으로 움직이거나, 매도 주도 체결대금 비율이 `sellImbalanceThreshold` 이상으로 커진다.
- 시간 만료: `maxHoldingSec` 안에 목표 수익에 도달하지 못하면 포지션을 축소하거나 청산 후보로 분류한다.
- 신호 반전: 매수 진입 근거였던 체결 불균형과 호가 지지가 사라지면 청산 후보로 분류한다.

시장가 매도는 체결 가능성이 높지만 스프레드와 슬리피지 비용이 커질 수 있다. 지정가 매도는 비용을 줄일 수 있지만 미체결 리스크가 있다. 따라서 초기 설계는 dry-run에서 둘 다 시뮬레이션하고, 실거래 전에는 주문 유형별 성과를 분리 집계한다.

## 기대수익 계산

틱 전략은 작은 변동폭을 노리므로 비용 계산이 우선이다.

필수 변수:

- `entryPrice`: 예상 진입 가격
- `exitPrice`: 예상 청산 가격
- `buyFeeRate`: 매수 수수료율
- `sellFeeRate`: 매도 수수료율
- `spreadCostRate`: 스프레드로 인한 비용률
- `slippageRate`: 체결 지연 및 호가 소진 예상 비용률
- `safetyMarginRate`: 전략이 요구하는 최소 추가 마진

진입 가능 조건:

```text
expectedGrossRate = (expectedExitPrice - expectedEntryPrice) / expectedEntryPrice
requiredRate = buyFeeRate + sellFeeRate + spreadCostRate + slippageRate + safetyMarginRate

enter only if expectedGrossRate > requiredRate
```

수수료율은 코드에 고정하지 않는다. 실거래 연결 전 `orders/chance` 응답 또는 사용자 설정에서 주입하고, 백테스트에서는 기간별 수수료율 설정을 별도로 둔다.

최소 주문 금액도 전략 파라미터가 아니라 주문 가능정보와 사용자 설정의 제약으로 처리한다. 주문 금액이 너무 작으면 수수료와 호가 단위 때문에 기대값이 왜곡되므로, 백테스트에는 최소 주문 금액보다 큰 별도 최소 전략 금액을 둔다.

## 포지션 크기와 리스크 제한

초기 전략은 단일 마켓 단일 포지션만 허용한다. 다중 마켓 동시 운용은 신호 검증 이후에 추가한다.

기본 제한:

- 마켓별 최대 보유 금액
- 전체 자동 전략 최대 노출 금액
- 1회 주문 최대 KRW
- 일일 누적 손실 한도
- 연속 손실 횟수 한도
- 일일 주문 횟수 한도
- 마켓별 쿨다운
- 급변장 회피용 전체 중단 조건

중단 조건:

- 일일 실현 손익이 `dailyStopLossRate` 이하로 내려간 경우
- 최근 `lossStreakLimit`회 거래가 모두 손실인 경우
- WebSocket 연결이 불안정하거나 체결/호가 데이터가 `staleDataMs` 이상 갱신되지 않은 경우
- Upbit 주문 API에서 인증 오류, 권한 오류, rate limit 오류가 반복되는 경우
- 스프레드가 평소 대비 급격히 확대된 경우

## 시장 상황별 동작

### 횡보장

스프레드와 수수료가 수익 폭을 대부분 가져갈 수 있다. 체결 불균형이 약하고 변동성이 낮으면 진입하지 않는다. 횡보장에서의 전략은 빈번한 거래보다 대기 상태가 기본이다.

### 급등장

매수 추격 신호가 많이 발생하지만 슬리피지가 커진다. 급등장에서는 시장가 진입을 제한하고, 호가 잔량이 충분하며 스프레드가 좁을 때만 후보로 둔다. 익절 목표는 빠르게 도달할 수 있지만 반전도 빠르므로 `maxHoldingSec`를 짧게 둔다.

### 급락장

매수 신호가 일시 반등처럼 보일 수 있다. 급락장에서는 신규 매수를 중단하거나, 상위 타임프레임 캔들 필터가 회복될 때까지 대기한다. 보유 포지션은 손절 조건을 우선한다.

### 저유동성 장

호가 잔량이 얇고 체결 간격이 길어 백테스트보다 실제 체결 결과가 나빠질 수 있다. 최근 N초 체결대금과 호가 잔량이 기준 미만이면 전략 대상에서 제외한다.

## 주문 방식 비교

### 시장가 매수와 시장가 매도

장점은 빠른 체결이다. 단점은 스프레드와 슬리피지를 직접 부담한다는 점이다. 틱 전략에서는 비용이 작지 않으므로 초기 실거래 후보에서는 보수적으로 사용한다.

### 지정가 주문

장점은 가격 통제가 가능하다는 점이다. 단점은 미체결과 부분 체결이 전략 결과를 크게 흔들 수 있다는 점이다. 지정가 주문을 쓰려면 미체결 취소, 재주문, 부분 체결 처리, 주문 상태 추적이 먼저 필요하다.

### 최유리 지정가

시장가보다 가격 통제가 있고 지정가보다 체결 가능성이 높을 수 있다. 다만 Upbit 주문 가능 유형과 `time_in_force` 지원 여부를 `orders/chance`로 확인해야 한다. 구현 전에는 dry-run과 주문 테스트 API로 별도 검증한다.

## 백테스트와 모의매매 계획

### 1단계: 오프라인 백테스트

- REST `trades/ticks`로 최근 체결 데이터를 수집한다.
- 가능하면 캔들과 ticker 데이터를 함께 저장해 시장 상황을 분류한다.
- 수수료, 스프레드, 슬리피지 가정을 반드시 적용한다.
- 각 거래의 진입 근거, 청산 근거, 비용 차감 전후 손익을 모두 기록한다.

주의할 점:

- REST tick 데이터만으로 실시간 호가 소진과 미체결을 정확히 재현하기 어렵다.
- 백테스트 결과는 실거래 성과로 간주하지 않는다.
- 지나치게 많은 파라미터 조합 탐색은 과최적화를 만들 수 있으므로 검증 기간을 분리한다.

### 2단계: 실시간 dry-run

- WebSocket trade/orderbook을 구독한다.
- 실제 주문 대신 가상 주문 이벤트를 만든다.
- 가상 진입 가격은 당시 최우선 호가와 슬리피지 가정으로 계산한다.
- 가상 청산도 동일한 비용 모델을 적용한다.
- 최소 1주 이상 마켓별 결과를 누적한다.

기록 항목:

- 마켓
- 신호 발생 시각
- 진입 후보 가격
- 청산 후보 가격
- 신호 점수
- 주문 방식 가정
- 수수료와 슬리피지 차감 전후 손익
- 체결 실패로 가정한 사례
- 중단 조건 발동 사례

### 3단계: 주문 테스트 API

실거래 전에 `POST /v1/orders/test` 또는 앱 내부 검증 경로를 사용한다. 주문 본문이 Upbit 규칙을 만족하는지 확인하고, 주문 가능정보 응답과 최소 주문 금액, 지원 주문 유형을 함께 검증한다.

### 4단계: 제한된 실거래

실거래는 별도 사용자 확인과 dry-run 해제 후에만 가능해야 한다. 초기에는 단일 마켓, 최소 전략 금액, 낮은 일일 주문 횟수, 엄격한 일일 손실 한도로 제한한다.

## 구현 시 필요한 코드 변경 범위

코드 구현은 이 문서의 범위가 아니지만, 이후 기능화하려면 다음 변경이 필요하다.

- Rust 백엔드에 `trade` 외 `orderbook` WebSocket 구독 추가
- React 상태에 틱 전략용 실시간 스냅샷 저장 구조 추가
- 전략 설정 모델 확장: 체결 불균형, 변동성, 쿨다운, 손절, 익절, 최대 보유 시간
- dry-run 전용 가상 체결 엔진 추가
- 전략 이벤트 로그와 결과 집계 화면 추가
- 주문 가능정보 기반 수수료율, 최소 주문 금액, 지원 주문 유형 검증 연결
- 실주문 전 `orders/test` 또는 내부 dry-run 통과 여부 저장
- 미체결 주문 조회와 취소 흐름 추가

입금과 출금 API는 이 전략 구현 범위에서 제외한다. API Key, JWT, 계정 정보, 주문 응답 원문은 로그와 UI에 그대로 노출하지 않는다.

## 실패 조건과 손실 가능성

이 전략은 다음 조건에서 실패할 수 있다.

- 체결 방향 불균형이 짧은 반등 또는 일시적 소음에 불과한 경우
- 급변장에서 주문 체결 가격이 예상보다 크게 밀리는 경우
- 지정가 주문이 체결되지 않아 기회가 사라지는 경우
- 부분 체결 후 반대 방향으로 가격이 움직이는 경우
- 수수료와 스프레드가 기대 변동폭보다 큰 경우
- WebSocket 지연 또는 끊김으로 오래된 데이터에 반응하는 경우
- 특정 기간 데이터에 과최적화되어 다른 시장 국면에서 성과가 무너지는 경우

따라서 실거래 연결 전에는 백테스트, 실시간 dry-run, 주문 테스트 API, 제한된 실거래 단계를 순서대로 통과해야 한다.
